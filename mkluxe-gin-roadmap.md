# Jewelry Store Backend Blueprint (Go + Gin)

This repository is the **Go backend** for a jewelry catalog website. The public storefront is handled by Next.js, while this backend provides product management, authentication, RBAC, inquiry handling, catalog import, and admin APIs.

The goal of this document is to give a coding agent a single source of truth so it can generate the backend **file by file** without repeatedly reading the whole codebase.

---

## 1) Product and platform scope

### What this backend must do
- Store and serve jewelry products for the public site.
- Support a single category per product, plus optional secondary categories only when clearly relevant.
- Support product variants, images, FAQs, SEO fields, and availability flags.
- Provide an admin API for create, edit, delete, publish/unpublish, and sorting.
- Support login, role-based access control, and audit logging.
- Support inquiry generation for WhatsApp-based selling.
- Support catalog import from structured JSON generated by an image-to-product extractor.

### What this backend does **not** do
- No cart.
- No payment gateway.
- No checkout flow.
- No order management system unless added later.

---

## 2) Recommended stack

### Core choices
- **Language:** Go
- **HTTP router:** Gin
- **API style:** JSON REST API
- **Database:** MongoDB
- **Auth:** JWT access + refresh tokens
- **RBAC:** role + permission middleware, optionally Casbin later
- **Admin UI:** Next.js frontend consuming admin endpoints
- **File/media handling:** cloud storage or local storage abstraction

### Why this stack fits
- Gin is popular for production Go APIs and keeps route handling straightforward.
- MongoDB fits product documents with flexible fields such as variants, FAQs, and specifications.
- Next.js is better suited than Go for the admin panel UI, while Go handles the data and permissions layer.

## 4) File-by-file blueprint

### Root files

#### `README.md`
This file is the main blueprint for the coding agent and human maintainers.

It explains:
- what the backend does
- what it does not do
- the stack used
- the folder structure
- the implementation phases
- the RBAC model
- the product import expectations

It does not export runtime code. It acts as the project contract.

#### `go.mod`, `go.sum`
These define and lock Go dependencies.

They import nothing directly in source code, but they determine which packages the project can use. Keep them stable and minimal.

#### `.env.example`
Contains sample environment variables such as:
- server host and port
- MongoDB URI and database name
- JWT secret and expiry values
- CORS origins
- media storage settings
- admin bootstrap values

It should not contain real secrets.

#### `.gitignore`
Keeps build artifacts, local environment files, and editor files out of Git.

#### `Dockerfile`
Builds the backend into a production container.

It should:
- copy the Go source
- download dependencies
- build the binary
- run the binary in a minimal final image

#### `docker-compose.yml`
Local development orchestration.

Usually runs:
- the Go API
- MongoDB
- optional MongoDB admin tool if needed

#### `Makefile`
Convenience command runner.

Suggested targets:
- `make run`
- `make test`
- `make build`
- `make lint`
- `make seed`
- `make docker-up`

## 20) Product import schema support

This backend should be ready to accept the structured product JSON produced by the image extractor.

### Import expectations
- One product has exactly one primary category.
- Secondary categories are optional and must still come from the approved category list.
- Variants, FAQs, images, and summary fields should be preserved.
- Unknown values should remain null or empty, not invented.
- Slug should be generated or normalized by the backend if missing.
- Admin import should support both:
  - create new product
  - update existing product by slug or external reference

### Import file behavior
The import service should:
- validate the incoming JSON
- map it into internal domain models
- reject unapproved category values
- reject malformed variants/images/FAQ blocks
- return a structured import result with counts and errors

## 21) RBAC design

### Roles
- **super_admin:** full access
- **admin:** manage products, categories, inquiries, users within limits
- **editor:** manage catalog content only
- **support:** view inquiries and basic catalog data

### Permission examples
- `products.read`
- `products.write`
- `products.publish`
- `categories.write`
- `users.write`
- `inquiries.read`
- `inquiries.update`
- `imports.run`
- `audit.read`

Permissions should be checked in middleware and/or service layer, not scattered randomly.

---

## 22) API response conventions

All endpoints should return predictable JSON.

### Success shape
- success flag
- message
- data
- meta when paginated

### Error shape
- success flag set to false
- error code
- human-readable message
- optional field errors

Keep this format consistent for the frontend and coding agent.

## 25) Coding style rules for the generated backend

- Keep handlers thin.
- Keep repositories database-only.
- Put business logic in services.
- Keep models clean and explicit.
- Prefer small, composable functions.
- Use consistent JSON tags.
- Use validated DTOs at the boundaries.
- Avoid overengineering the first version.
- Build the admin panel as a separate Next.js app that consumes these APIs.

## 26) Final target architecture

The finished system should look like this:

- **Next.js public site** for customers
- **Next.js admin panel** for internal management
- **Go + Gin REST API** for all data, auth, RBAC, and business rules
- **MongoDB** for product and admin data
- **WhatsApp inquiry flow** instead of checkout

That gives you a clean, professional backend that is simple enough to maintain and strong enough to grow later.
